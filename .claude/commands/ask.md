## Spec Interview

Глибоке інтерв'ю по специфікації для виявлення прогалин, ризиків та неоднозначностей.

**Input:** $ARGUMENTS (шлях до SPEC.md)
**Output:** SPEC-refined.md поруч з оригіналом

## Process

### Step 1 — READ

Прочитай файл `$ARGUMENTS`. Зрозумій скоуп фічі, основні вимоги, та контекст.

### Step 2 — ANALYZE

Перед тим як ставити питання, проаналізуй спек і визнач:
- Що НЕ специфіковано (implicit assumptions)
- Що неоднозначно (можна прочитати двома способами)
- Де є технічні ризики (складні інтеграції, edge cases)
- Де бракує деталей (error states, concurrency, data flow)

### Step 3 — INTERVIEW

Використовуй AskUserQuestion для глибокого інтерв'ю. По 2-3 питання за раз.

**Категорії питань (чергуй):**
- **Technical:** архітектура, data flow, edge cases, error handling, API contracts, concurrency
- **UI/UX:** стани компонентів, transitions, empty/error/loading states, що бачить юзер коли щось пішло не так
- **Tradeoffs:** performance vs readability, complexity vs flexibility, scope vs deadline
- **Integration:** з існуючим кодом, breaking changes, міграції, backwards compatibility
- **Concerns:** що може піти не так в проді, масштабування, maintenance burden

**Правила:**
- Питання мають бути НЕ очевидні. Не "який колір кнопки?", а "що станеться коли юзер відкриє 3 сесії одночасно?"
- Кожне питання має розкривати те, що автор спеку міг не врахувати
- Продовжуй доки не закриєш всі знайдені прогалини
- Групуй питання логічно — пов'язані теми разом
- Якщо відповідь породжує нове питання — задай його в наступному раунді

### Step 4 — WRITE

Створи файл `SPEC-refined.md` в тій самій директорії що й оригінал.

Структура:
1. Оригінальний скоуп (коротко)
2. Уточнені вимоги (з відповідей інтерв'ю)
3. Технічні рішення (що вирішили під час інтерв'ю)
4. Edge cases та обмеження
5. Відкриті питання (якщо залишились)

Покажи шлях до створеного файлу.
